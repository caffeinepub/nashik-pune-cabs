1. PROJECT OVERVIEW
================================================================================
This is a cab booking application for Nashik to Pune taxi service built on
the Internet Computer platform. It features:
- Public booking interface for customers
- Admin dashboard for viewing all bookings
- Internet Identity authentication
- Role-based access control (admin/user/guest)

Technology Stack:
- Backend: Motoko (Internet Computer)
- Frontend: React + TypeScript + Tailwind CSS
- State Management: React Query (@tanstack/react-query)
- Authentication: Internet Identity

4. ROUTING & NAVIGATION
================================================================================

ROUTING ARCHITECTURE
--------------------
Type: Simple hash-based routing (no external router library)
Implementation: frontend/src/App.tsx

Routes:
┌─────────────────┬──────────────────────────────────────────────────────┐
│ URL Pattern     │ Component/View                                       │
├─────────────────┼──────────────────────────────────────────────────────┤
│ /               │ Public landing page with booking form                │
│ #/              │ Public landing page with booking form                │
│ #/admin         │ Admin bookings dashboard (requires authentication)   │
└─────────────────┴──────────────────────────────────────────────────────┘

Navigation Implementation:
- Uses window.location.hash to determine current route
- Updates via window.location.hash = '#/admin' or '#/'
- No <Link> components needed, uses standard <a href="#/admin">

Access Control:
- Public routes: Accessible to all users (guest, user, admin)
- Admin routes: Require authentication + admin role verification

All methods are called through the actor initialized in useActor.ts

PUBLIC METHODS (accessible to all users including guests)
----------------------------------------------------------
createBooking(name: string, phone: string): Promise<string>
├── Purpose: Create a new booking
├── Parameters:
│   ├── name: Customer name (2-30 chars)
│   └── phone: Phone number (10-15 chars, digits/+ only)
├── Returns: String with format "{bookingId} {name} {phone}"
├── Frontend Hook: useCreateBooking (hooks/useCreateBooking.ts)
└── Used By: BookingForm component

├── Frontend Hook: useBookingsWithIds (hooks/useBookingsWithIds.ts)

ERROR HANDLING
--------------
Backend errors are thrown via Runtime.trap() with descriptive messages:
- "Unauthorized: Only admins can view all bookings"
- "Unauthorized: Admin already initialized"
- "Unauthorized: Anonymous principals cannot be admin"
- "Name too short" / "Name too long"
- "Phone number too short" / "Phone number too long"
- "Phone number contains invalid characters"

Frontend handles these via React Query error states and displays
appropriate error messages to users.

7. DATA FLOW
================================================================================

BOOKING CREATION FLOW
----------------------
1. User fills BookingForm on landing page
2. Client-side validation checks all fields
3. Form submits → useCreateBooking mutation
4. Hook calls actor.createBooking(name, phone)
5. Backend validates, generates ID, stores booking
6. Backend returns: "{bookingId} {name} {phone}"
7. Hook extracts bookingId from response
8. Hook invalidates 'bookings-with-ids' query cache
9. BookingConfirmation component displays success with ID

ADMIN ACCESS FLOW
-----------------
1. User navigates to #/admin
2. AdminBookingsPage checks authentication via useInternetIdentity
3. If not logged in → Show login prompt
4. User clicks login → Internet Identity flow
5. After login, check hasAdmin() via useHasAdmin hook
6. If no admin exists → Show "Claim Admin Access" with principal ID
7. User clicks "Claim Admin Access" → useInitializeAdmin mutation
8. Backend sets caller as admin (if first time)
9. Check isCallerAdmin() via useIsCallerAdmin hook
10. If admin → Show NotificationsAdminView
11. If not admin → Show "Access Denied"

BOOKINGS DISPLAY FLOW (Admin)
------------------------------
1. NotificationsAdminView mounts
2. useBookingsWithIds hook activates
3. Hook calls actor.getAllBookingsWithIdsSorted()
4. Backend checks if caller is admin (throws if not)
5. Backend returns array of [id, booking] tuples sorted newest-first
6. React Query caches result with key 'bookings-with-ids'
7. Component renders table with booking data
8. User clicks refresh → Query refetches from backend
9. Cache invalidation happens on new booking creation

AUTHENTICATION STATE FLOW
--------------------------
1. useInternetIdentity hook initializes on app load
2. Checks for stored identity in IndexedDB
3. If found → Restores session (identity available)
4. If not found → User is anonymous (identity = null)
5. Login → Opens Internet Identity window
6. User authenticates → Identity stored
7. Logout (clear()) → Clears identity and cache
8. All queries using actor are disabled when identity is null

NOTES
================================================================================

ADMIN INITIALIZATION
--------------------
- First authenticated user can claim admin access
- Only one admin can be initialized (adminInitialized flag)
- Anonymous principals cannot become admin
- Admin status persists across canister upgrades

AUTHORIZATION PATTERN
---------------------
- Backend enforces all authorization via Runtime.trap()
- Frontend checks are for UX only (show/hide UI elements)
- Never rely on frontend checks for security
- Always handle authorization errors gracefully in UI

CACHING STRATEGY
----------------
- React Query caches all backend responses
- Query keys: 'bookings-with-ids', 'isCallerAdmin', 'hasAdmin', etc.
- Mutations invalidate related queries automatically
- Cache cleared on logout via queryClient.clear()
